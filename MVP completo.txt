📘 DOCUMENTO MAESTRO: PROYECTO LOCAL
Plataforma SaaS de Autonomía para Restaurantes
Versión: MVP Fase 0-1
Stack: Node.js + Fastify + Next.js + PostgreSQL + Prisma + ManyChat
Fecha: Abril 2026
1. CONCEPTO CENTRAL
LOCAL es el "Shopify para restaurantes" — no un marketplace. Cada restaurante tiene su propio menú QR independiente, sin comisiones del 30% ni intermediarios en pagos.
Filosofía "Autonomía Total":
Dueño controla cuándo abrir/cerrar/pausar (estados: OPEN/BUSY/CLOSED)
Pagos P2P: Cliente paga directo al restaurante (efectivo/transferencia)
WhatsApp como interfaz operativa: Aceptar/Rechazar pedidos con botones
Sin app del cliente: PWA web accesible por QR
Flujo Básico:
Cliente escanea QR → Ve menú → Agrega carrito → Checkout (nombre+teléfono)
Sistema crea orden #001 (status: PENDING) → Webhook a ManyChat
ManyChat envía WhatsApp al dueño: "Pedido #001 - 2 tacos C$300 [ACEPTAR][RECHAZAR]"
Dueño toca botón → API actualiza estado → Cliente ve confirmación en web

// packages/database/prisma/schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model Restaurant {
  id           String   @id @default(uuid())
  slug         String   @unique  // "casa-antigua"
  name         String
  whatsapp     String   // Número de ManyChat
  isOpen       Boolean  @default(true)
  currency     String   @default("NIO")
  createdAt    DateTime @default(now())
  
  categories   Category[]
  orders       Order[]
}

model Category {
  id            String   @id @default(uuid())
  restaurantId  String
  restaurant    Restaurant @relation(fields: [restaurantId], references: [id])
  
  name          String   // "Entradas", "Sushi"
  sortOrder     Int      @default(0)
  
  items         MenuItem[]
  
  @@index([restaurantId])
}

model MenuItem {
  id            String   @id @default(uuid())
  restaurantId  String
  categoryId    String
  category      Category @relation(fields: [categoryId], references: [id])
  
  sku           String?  // Código interno (EN-01, S-CL-01)
  name          String
  description   String?
  price         Decimal  @db.Decimal(10,2)
  isAvailable   Boolean  @default(true)
  imageUrl      String?
  
  @@index([restaurantId, categoryId])
  @@index([isAvailable])
}

model Order {
  id                String   @id @default(uuid())
  restaurantId      String
  restaurant        Restaurant @relation(fields: [restaurantId], references: [id])
  
  orderNumber       Int      // Autoincremental por restaurante (001, 002...)
  customerName      String
  customerPhone     String
  status            OrderStatus @default(PENDING)
  
  total             Decimal  @db.Decimal(10,2)
  
  createdAt         DateTime @default(now())
  acceptedAt        DateTime?
  readyAt           DateTime?
  
  items             OrderItem[]
  
  // Tracking ManyChat
  manyChatMessageId String?
  customerNotified  Boolean @default(false)
  
  @@unique([restaurantId, orderNumber])
  @@index([restaurantId, status])
  @@index([restaurantId, createdAt])
}

model OrderItem {
  id            String   @id @default(uuid())
  orderId       String
  order         Order    @relation(fields: [orderId], references: [id], onDelete: Cascade)
  
  menuItemId    String?
  itemName      String   // Snapshot del nombre (por si cambia después)
  itemPrice     Decimal  @db.Decimal(10,2) // Snapshot del precio
  quantity      Int
  notes         String?
  
  @@index([orderId])
}

enum OrderStatus {
  PENDING
  ACCEPTED
  READY
  COMPLETED
  REJECTED
  CANCELLED
}

8. REGLAS DE ORO (Anti-VibeCoding)
No comisiones, no pasarelas de pago en MVP: Siempre P2P (efectivo/transferencia). El dueño marca "Completado" manualmente cuando recibe pago.
No autenticación compleja en Fase 0: Magic links por email para dashboard, o simplemente sesión hardcodeada para tu restaurante mientras haces dogfooding.
No edits complejos de menú en MVP: Si necesitas cambiar un precio, usa Prisma Studio o reinicia el seed. El editor visual es Fase 4.
Validación de precios: Backend SIEMPRE recalcula el total de la orden sumando MenuItem.price * quantity. Nunca confíes en el total que envía el frontend.
Una sola DB, múltiples tenants: Todas las órdenes en tabla orders, filtradas estrictamente por restaurant_id. Nunca una DB por restaurante.
